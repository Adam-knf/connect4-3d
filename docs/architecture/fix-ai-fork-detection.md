### 1.3 为什么 depth=4 看不到 fork（更新分析）

```
Black2→White2→Black3→White3→Leaf
                              ↑ ply 4: White3 堵了一条 2 连
                                Leaf 只看 1 条 2 连 → 低分

Black4 的杀招在 ply 5: 搜索树外!

InternalNode 试图在 ply 2 (Black3) 预支 +80 分
但 White2 的完美防守能让 Black3 的任何位置都不形成收敛 fork
→ InternalNode 最多给 +20 (1 条有扩展的 2 连)
→ 信号不足以克服中心位置偏差

根本原因:
  (3,1,0) 位于棋盘边缘 (x=3)，Minimax 叶节点评估天然劣势
  (1,1,0) / (2,1,0) 靠近中心，叶节点评估天然高分
  +80 的内部加分不足以扭转这个差距
```

## 二、Route C 设计方案（实现并验证）

### 2.1 核心思路

从"全有或全无 (conv true/false)"改为**连续信号**。

在 depth=4 约束下，AI 无法 100% 正确识别远期 fork。目标改为：
1. ✅ 修正评估中的错误（非连续计数）
2. ✅ 给搜索树提供持续、可靠的 fork 信号（非收敛依赖）
3. ✅ 消除 Layer3 的误导性偏差
4. ❌ 让 (3,1,0) 和 (1,1,0) 的评分持平 — 未达成，depth=4 本质限制

### 2.3 修改1：非连续过滤（已验证）

**问题**：`evaluateInternalNode` 的过滤条件 `aiCount >= 2 && oppCount === 0 && openEnds > 0` 仅检查棋子**数量**，不检查**连续性**。

**修复方案**：
- 移除 `oppCount === 0` 条件：对手棋子可能在 segment 端点，不影响 2 连块连续性
- 移除 `openEnds > 0` 条件：openEnds 只反映 segment 两端，可能遗漏中段扩展点
- 依赖 `findExtensionCells` 作为最终验证器：验证连续性 + 扩展点可下性

**验证结果**：`findExtensionCells` 正确过滤了非连续的线，对反斜线 2 连块正确找到扩展点 (1,3,0)。

### 2.4 修改2：扩展点递进加分（已验证）

**评分规则**：

```
有扩展点的 2 连线数 | 加分      | 说明
-------------------|----------|------
0                  | 0        | 无威胁
1                  | TWO_OWN=20 | 单条 2 连
2+                 | TWO_OWN×4=80 | 多条威胁线，上限 80
```

**重要修正**：v3.0 设计中原定 "3+ → 300"，实现验证后发现：
- `aiLinesWithExt` 可能因重叠段计数偏高（最多到 6-8 条）
- 300 分加在 Minimax 内部节点会过度放大，导致中心位置获得不成比例的优势
- 改为 2+ 统一给 80 分，作为"有 fork 潜力"的信号而非必胜判断

### 2.5 修改3：Layer3 降权（已验证）

```
Minimax 深度 | 原系数  | 新系数   | 理由
-------------|--------|---------|------
depth=1      | 1.0    | 1.0     | 无 Minimax，Layer3 是主要信号
depth=2      | 1.0    | 0.5     | Minimax 浅，Layer3 辅助
depth=3 (MEDIUM) | 0.3 | 0.05   | Minimax 已提供足够信息
depth=4 (HARD)   | 0.3 | 0.05   | Minimax depth=3 覆盖叉子动态
```

**降权到 0.05 而非 0.1 的原因**：实测发现 0.1 时 Layer3 仍贡献 600×0.1=60 分给 (1,1) 位置，偏差过大。

## 三、搜索扩展方案（尝试并回退）

### 3.1 方案描述

在 Minimax AI 回合检测到 2+ 威胁线时，扩展 1 ply：

```
普通搜索: Black2(d=3)→White2(d=2)→Black3(d=1)→White3(d=0)→Leaf
扩展搜索: Black2(d=3)→White2(d=2)→Black3(d=2)→White3(d=1)→Black4(d=0)→Leaf
```

### 3.2 回退原因

1. **过放大**：扩展让 (2,1,0) 获得 814 分 vs (3,1,0) 的 596 分，因为 (2,1) 有 3 条威胁线 (aiLinesWithExt=3)，内部加分 80 配合扩展树让中心位置过度受益
2. **不均匀受益**：扩展对所有位置一视同仁，但中心位置的叶节点评估天然更高 → 扩展放大的是已有的偏差
3. **选择性偏差**：fork 扩展无法区分"真 fork"和"假阳性 fork"，任何 2+ 威胁线都会触发

### 3.3 教训

搜索扩展只在"扩展的目标明确且均匀受益"时才有效。当 fork 位置在棋盘边缘时，扩展反而会放大中心位置的已有优势。

## 四、最终评估

### 4.1 修复效果

| 指标 | 修复前 | 修复后 | 评估 |
|------|--------|--------|------|
| 非连续计数 | 假阳性 | 正确过滤 | ✅ |
| 收敛检测 | 二值 (conv yes/no) | 连续评分 | ✅ |
| Layer3 偏差 | 180 vs 45 | ~30 vs ~8 | ✅ 大幅降低 |
| (3,1,0) 选择率 | 0% | 0% | ❌ 未改善 |
| 节点数 | ~50855 | ~53920 | ✅ 可控 |

### 4.2 已修复的问题

代码质量层面的改进（即使 fork 检测未达预期）：

1. **evaluateInternalNode** 现在正确使用 `findExtensionCells` 验证连续性，不再依赖 `openEnds` 和 `oppCount` 等表面条件
2. **内部加分上限 80** 防止了多条威胁线导致的评分爆炸
3. **Layer3 系数 0.05** 消除了静态评估对 Minimax 的过度干扰
4. 去重的 `seenAiKeys` / `seenOppKeys` 机制配合扩展点计数，避免物理线重叠段的重复计数

### 4.3 本质限制

```
depth=4 的搜索深度限制:

Black 的最佳 fork 路线:
  Black1(2,2) → White1(1,2) → Black2(3,1) → White2(?,?) → Black3(1,1) → White3(?,?) → Black4(?,?) 杀招
  ply 0        ply 1        ply 2        ply 3          ply 4        ply 5          ply 6

在 depth=4 时，搜索树在 White3 (ply 4/5) 结束，Black4 看不到。
任何 InternalNode 加分都只是信号，无法替代实际搜索。

结论: 只有 depth=6 (ply 6+1) 才能可靠看到这个 fork。
```

### 4.4 改进建议（未来）

如果未来需要真正修复此问题，可行方向：

| 方案 | 描述 | 代价 |
|------|------|------|
| depth=6 | HARD 搜索深度提升到 6 | 节点数 ~50000×100=500万，超时不可行 |
| 选择性扩展 | 仅在检测到 2+ 威胁线时扩展 1 ply，且只对 AI 回合 | 如前所述，有选择性偏差风险 |
| **静态 fork 检测** | 在 `evaluateMove` 层直接检测 fork 模式，不依赖 Minimax 搜索 | 最稳妥，复杂度在评估函数而非搜索树 |
| 迭代加深 | 时间允许时自动加深 | 需设置合理时间预算 |

**推荐方案**：静态 fork 检测。在 `evaluateMove` 中直接判断：如果某位置落子后 AI 有 2+ 条有扩展点的 2 连，且对手一步只能堵一条，直接给予高分。这避免了搜索深度限制。
